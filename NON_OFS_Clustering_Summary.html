1. Executive Summary
This analysis clusters every NON-OFS address in the Verizon ILEC footprint — 12.26 million addresses representing 15.9 million premises across 2,078 wire centers — into 68,071 geographically contiguous, build-ready hubs using road-graph connectivity and a four-component financial quality model. Each hub is then scored on two independent axes — Urgency (obligation pressure, copper retirement imminence, dispatch frequency) and Value (IRR, penetration, revenue potential) — producing a 2×2 Eisenhower prioritization matrix. The result is a single, repeatable framework for answering "where should we build next?" by balancing financial return with regulatory and operational urgency. Obligation tagging is decoupled from cluster formation so that monthly score refreshes never require re-clustering.

4. Formation Scoring

Each address receives a composite financial quality score (0–100) that determines seed priority during cluster formation and growth order during hub expansion.

Component Weight What It Measures Normalization
IRR V2 40% 15-year internal rate of return from V2 penetration ramp model Min-max within wire center; higher = better
Copper Salvage 25% Estimated recoverable value of existing copper plant copper_circuits × $200/circuit, normalized to WC max
BI Priority Rank 20% BI composite rank (dispatch, sales propensity, geographic synergy) Inverted min-max (lower rank = higher score)
Unit Density 15% MDU/MTU density bonus min(units / 10, 1.0) — caps at 10+ units
Design principle: IRR dominates because it directly measures build economics. Copper salvage is second because it represents both financial recovery and an operational imperative. BI rank captures operational synergies not embedded in IRR. Density rewards MDU/MTU cost efficiency.

Why obligations are excluded from formation: Obligations change (new mandates, date shifts, scope changes) but cluster boundaries should remain stable. Separating obligation tagging from formation means monthly score refreshes never require re-clustering.
Clustering Algorithm (6 Phases)
PhaseStepDescription
1Formation ScoringScore every address using the 4-component formula above
2Road-Group ConstructionSnap addresses to nearest TIGER road segment; group by connected road segments
3Adjacency GraphBuild graph of physically adjacent road groups (within morphology-specific radius caps)
4Seed & GrowHighest-scored road groups become hub seeds; adjacent groups merge via growth scoring (financial 35%, proximity 25%, IRR gravity 25%, street affinity 15%)
5Size EnforcementSplit oversized hubs, merge undersized ones. Targets: 150–450 units suburban, 400–700 urban, 50–200 rural
6FinalizeAssign hub IDs ({CLLI}_H{nnn}), compute centroids, build output records

6. Eisenhower Scoring

Each cluster is scored on two independent 0–100 axes, then assigned to a quadrant based on a threshold of 50 on each axis.

Urgency Score (0–100) — "How pressing is the need to build?"
Component Weight What It Measures Scoring
Obligation Tier 50% Highest obligation bucket present in the cluster COP_2026=100, COP_2027=80, COP_FUTURE=60, SBB=70, NSI=50, LFA=40, DISC=0
Copper Recycling Imminence 30% % of addresses with copper retirement date in 2026–2027 Linear: 0% → 0, 100% → 100
Dispatch Frequency 20% Average truck rolls per address in past 12 months min(avg_dispatch × 20, 100) — 5+ dispatches/yr = 100
Value Score (0–100) — "How financially attractive is this cluster?"
Component Weight What It Measures Scoring
IRR V2 50% Average 15-year IRR across cluster 0% → 0, 15% → 50, 30%+ → 100 (linear)
Terminal Penetration 25% Average projected FiOS take rate at steady state min(pen × 200, 100) — 50%+ terminal = 100
Revenue Potential 25% Average annual EBITDA per address min(ebitda / $500, 100) — $500+/yr = 100

9. Monthly Refresh Process

Run monthly_refresh.py to update scores without re-clustering. Cluster boundaries remain stable — only scores and tags change.

  1. Re-extract address data — Pull updated BI scores from Oracle (IRR, EBITDA, dispatch, copper counts)
  2. Re-tag obligations — Apply updated copper recycling dates, SBB flags, and NSI matches
  3. Re-score Eisenhower matrix — Recompute urgency and value scores for each cluster
  4. Regenerate polygon metadata — Update attributes on existing geometry (geometry unchanged)
  5. Write Oracle tables — Insert new RUN_DATE partition into CLUSTER_ASSIGNMENT_MASTER and CLUSTER_SUMMARY_MASTER
Dry run: Use --dry-run to score without writing to Oracle. Useful for validation before production refresh.
